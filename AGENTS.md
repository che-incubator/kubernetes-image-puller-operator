# Agent Guidelines

This file provides context for AI coding agents working on the Kubernetes Image Puller Operator. It complements `CONTRIBUTING.md` and `README.md` with non-obvious project knowledge that is difficult to derive from code alone.

## Quick Reference

```bash
make build          # Build operator binary
make test           # Run unit tests (requires setup-envtest)
make manifests      # Regenerate CRDs and RBAC from markers
make generate       # Regenerate DeepCopy methods
make bundle         # Regenerate OLM bundle (requires python-yq)
make gen-deployment # Regenerate deploy/deployment/ manifests
make docker-build   # Build container image
```

Use `VERBOSE=1` with any target to see commands — output is silent by default.

## Architecture

The operator manages a `KubernetesImagePuller` custom resource (`apiVersion` `che.eclipse.org/v1alpha1`; group `che.eclipse.org`, version `v1alpha1`). When a CR is created, the operator provisions a ConfigMap, Deployment, ServiceAccount, Role, and RoleBinding for the downstream [kubernetes-image-puller](https://github.com/che-incubator/kubernetes-image-puller), which runs a DaemonSet to pre-pull container images onto cluster nodes.

The CR spec fields map to ConfigMap environment variables that the image puller reads. When modifying the CRD spec, the corresponding ConfigMap key must also be added in `pkg/config/configmap.go` and tested in `controllers/kubernetesimagepuller_controller_test.go`.

A validating webhook enforces that only one `KubernetesImagePuller` resource may exist per namespace.

### Key Directories

| Path | Purpose |
|------|---------|
| `api/v1alpha1/` | CRD types, webhook, generated deepcopy |
| `controllers/` | Reconciliation logic and tests |
| `pkg/config/` | ConfigMap creation and defaults |
| `pkg/rbac/` | RBAC resource creation |
| `config/` | Kustomize overlays (default, kubernetes, openshift, OLM) |

| `deploy/deployment/` | Generated deployment manifests — **do not hand-edit** |
| `build/` | Dockerfile, entrypoint, deploy.mk |
| `hack/` | License header templates |

## Critical Gotchas

### Build System

- **Python yq required for `make bundle`**: The bundle target uses `yq -riY`, which is specific to the Python `yq` wrapper (installed via `pip install yq`). The Go-based `yq` (Mike Farah) does not support the `-Y` flag. If `make bundle` fails to patch images in the CSV, fix them manually.

- **`.ONESHELL` is enabled**: Each Makefile recipe runs as a single shell script. Multi-line recipes share shell variables across lines. Some targets explicitly set `SHELL := /bin/bash` for bash features.

- **`setup-envtest` version sensitivity**: The `download-envtest` target installs `setup-envtest@latest`, which may require a newer Go than the project uses (Go 1.22). If this breaks, pin to a compatible release branch (e.g. `@release-0.19` for Go 1.22).

- **Tools install locally**: All build tools (controller-gen, kustomize, operator-sdk, opm) install into `./bin/`, not globally. The `bin/` directory is gitignored.

### Generated Files

- **`deploy/deployment/`**: Generated by `make gen-deployment`. Must be regenerated after any kustomize config changes. Uses `csplit` (GNU coreutils) internally.

- **`api/v1alpha1/zz_generated.deepcopy.go`**: Generated by `make generate`. Do not edit.

- **CRD YAML post-processing**: `make manifests` strips YAML document delimiters (`---`) from the generated CRD because they break OLM catalog source images.

- **License headers**: The `addlicense` tool must be run on generated files. The `manifests` and `bundle` targets handle this automatically.

### Versioning

Version must be synchronised in three places:

1. `Makefile` — `VERSION ?= x.y.z`
2. `version/version.go` — hardcoded string
3. `config/manager/kustomization.yaml` — kustomize `images` transformer `newTag`

The `make-release.sh` script automates this for releases.

### Naming

The OLM bundle package name is `kubernetes-imagepuller-operator` (no hyphen between "image" and "puller"), while the repository is `kubernetes-image-puller-operator`. This is a historical inconsistency — do not "fix" it.

### CRD Quirks

- JSON tag casing is inconsistent: the Go fields `CachingCpuRequest` and `CachingCpuLimit` have JSON tags `cachingCPURequest` and `cachingCPULimit` (capital CPU). CR YAML must use the JSON tag casing.

- The manager sets up a cache that filters by label `app=kubernetes-image-puller`. The controller can only observe resources with this label.

### Kustomize

- The `$(CONTROLLER_SERVICE_ACCOUNT)` variable in `config/manager/manager.yaml` is resolved via a `replacements` block in `config/default/kustomization.yaml`, sourcing the ServiceAccount name from `config/manager/service_account.yaml`.

- `config/openshift/olm/prefixed/kustomization.yaml` uses JSON patches with hardcoded array indices to remove cert/volume mounts. Update these indices if containers or volumeMounts are added or removed.

- `config/kubernetes/` requires cert-manager for webhook TLS. `config/openshift/` uses OpenShift's service-serving-cert-signer instead.

## CI Workflows

| Workflow | Trigger | Purpose |
|----------|---------|---------|
| `pr-check.yml` | Pull requests | Unit tests, image build, OLM scorecard |
| `next-build.yml` | Push to `main` | Build and push `:next` image to quay.io |
| `release.yml` | Manual dispatch | Tagged release (requires Python yq via pip) |

The Docker build runs `make test` inside the image build by default. The PR check workflow passes `SKIP_TESTS=true` to the Docker build since it runs tests separately.

## Contribution Workflow

- Fork-based contributions — do not push directly to the upstream repo
- Eclipse Contributor Agreement (ECA) and DCO sign-off required on all commits
- One concern per PR, link related issues
- Squash commits into clean, atomic units before requesting review
- Run `make test` before submitting
- Run `make gen-deployment` if kustomize configs were changed
